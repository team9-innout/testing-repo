### Rules

**Note:** The source ADR does not contain explicit R-PREFIX-### rule identifiers. The following rules are extracted from the Policies and Instructions sections:

- **MUST:** Implement a standardized configuration and environment management pattern that is accessible across all architectural layers (server-side routes, data models, client-side components, and hooks).
- **MUST:** Provide consistent access to environment variables, runtime configuration, and context-aware settings throughout the application stack.
- **MUST:** Use a centralized configuration approach that can be consumed by server components (routes, models), client components (UI fields, form components), and utility functions (custom hooks).
- **MUST:** Avoid coupling components to specific environment detection mechanisms.
- **SHOULD:** Separate concerns - components focus on behavior while configuration provides context.
- **SHOULD:** Design for testability by ensuring configuration can be mocked or injected consistently across different component types.
- **SHOULD:** Enable feature flags and environment-specific behavior without hardcoding values throughout the codebase.
- **SHOULD:** Govern and document configuration to prevent configuration bloat.
- **MAY:** Consider what should be configured vs. what should be hardcoded on a case-by-case basis.

### Verify

**Note:** The source ADR does not contain an explicit verify section with bash commands.

Suggested verification approach:

```bash
# Check for hardcoded environment-specific values
grep -r "process.env" --include="*.ts" --include="*.tsx" --include="*.js" --include="*.jsx" . | grep -v "config" | wc -l

# Verify centralized configuration exists
find . -name "config.*" -o -name "*config.ts" -o -name "*config.js"

# Check for direct environment checks in components
grep -r "NODE_ENV" --include="*.tsx" --include="*.jsx" . | grep -v "config" | wc -l
```

**Accept when:**
- Configuration access pattern is consistent across server-side routes, data models, client-side components, and custom hooks
- Centralized configuration management is in place and documented
- Components are decoupled from specific environment detection mechanisms
- Environment-specific settings are accessible without hardcoding throughout the codebase
- Configuration structure supports testability through mocking or injection
- Feature flags and environment-specific behavior use the centralized configuration approach
