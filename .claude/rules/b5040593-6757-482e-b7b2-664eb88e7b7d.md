<rule_activation id="b5040593-6757-482e-b7b2-664eb88e7b7d" title="Adopt Centralized Test Helper and Mocking Infrastructure for Component Testing" applies_to="packages/core/**">
These rules are ALWAYS ACTIVE for all files matching `packages/core/**`.

The codebase requires consistent testing patterns across multiple core packages including component resolution, UI components (Puck), reducer actions, and migration logic. This rule establishes centralized test helper and mocking infrastructure to reduce duplication and improve test maintainability.
</rule_activation>

### Rules

- **R-TEST-001** MUST: Create reusable test helpers in `__helpers__/index.tsx` for common test setup and teardown operations when writing tests for core package components.

- **R-TEST-002** MUST: Use standardized mock implementations for component data resolution across all test suites to ensure consistency.

- **R-TEST-003** MUST: Follow the `.spec.tsx` naming convention for testing React components with TypeScript.

- **R-TEST-004** SHOULD: Import shared fixtures and utilities from centralized helper modules rather than duplicating test setup code across test files.

- **R-TEST-005** SHOULD: Organize test infrastructure using dedicated `__helpers__` and `__tests__` directories with shared utilities.

- **R-TEST-006** MAY: Design helpers with proper separation of concerns to avoid coupling between tests.

### Verify

**Note:** The source ADR does not provide explicit verification commands. The following checks are recommended:

```bash
# Check for test helper infrastructure
find packages/core -type d -name "__helpers__" | head -5

# Verify .spec.tsx convention usage
find packages/core -name "*.spec.tsx" | wc -l

# Check for centralized test utilities
find packages/core -path "*/__helpers__/index.tsx" | head -5
```

**Accept when:**
- Test code duplication is reduced across test files through reusable utilities
- Common testing patterns are centralized in `__helpers__` directories
- React component tests follow `.spec.tsx` naming convention
- Shared fixtures and utilities are available for import across test suites
- Mock implementations for component data resolution are standardized
- Test infrastructure supports component-driven architecture testing, state management validation, and data transformation verification

Claude Code MUST NOT skip or defer verification. When writing or modifying tests in packages/core/**, always check for existing test helpers and reuse them before creating new test utilities.
