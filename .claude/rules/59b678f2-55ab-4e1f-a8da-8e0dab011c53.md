<rule_activation id="59b678f2-55ab-4e1f-a8da-8e0dab011c53" title="Adopt Defensive Input Validation and Boundary Checking for Configuration and Environment Management" applies_to="**/*">
These rules are ALWAYS ACTIVE for all files. Apply defensive input validation and boundary checking at all configuration and environment management touchpoints across routes, models, UI components, and utility libraries.
</rule_activation>

### Rules

- **R-DIV-001** MUST: Validate environment variables and configuration values before use. Ensure all environment variables and configuration values are checked for existence, type, and validity before being used in application logic.

- **R-DIV-002** MUST: Implement type guards and runtime checks for dynamic inputs. Add runtime type checking for all dynamic inputs that cannot be statically verified at compile time.

- **R-DIV-003** MUST: Provide fallback values for missing or invalid configuration. Define sensible defaults and fallback values for optional configuration to enable graceful degradation.

- **R-DIV-004** MUST: Sanitize user inputs in form fields and UI components. Apply input sanitization at all user input boundaries including form fields, query parameters, and interactive UI components.

- **R-DIV-005** MUST: Establish clear contracts at module boundaries where configuration data flows. Define explicit interfaces and validation rules at boundaries between server-side routes, data models, client-side hooks, and UI components.

- **R-DIV-006** SHOULD: Apply validation logic consistently across all architectural layers. Ensure validation patterns are uniform across server-side routes, data models, client-side hooks, and UI components.

- **R-DIV-007** SHOULD: Provide clear error messages and debugging information when configuration issues occur. Include context about what validation failed and what the expected values are.

- **R-DIV-008** SHOULD: Make configuration dependencies explicit and self-documenting through validation logic. Use validation code as documentation of configuration requirements.

### Verify

**Note:** The source ADR did not include explicit verification commands. Consider implementing the following verification approaches:

```bash
# Search for unvalidated environment variable access
grep -r 'process\.env\.' --include="*.ts" --include="*.tsx" --include="*.js" --include="*.jsx" | grep -v -E '(validate|check|guard|fallback|default)'

# Search for direct configuration access without validation
grep -r 'config\[' --include="*.ts" --include="*.tsx" | grep -v -E '(validate|check|guard|\?\.|\?\?)'

# Look for form inputs without sanitization
grep -r '<input' --include="*.tsx" --include="*.jsx" | grep -v -E '(sanitize|validate|onChange)'

# Check for missing type guards on dynamic inputs
grep -r 'as any' --include="*.ts" --include="*.tsx"
```

**Accept when:**
- All environment variables and configuration values are validated before use with type guards or runtime checks
- Fallback values or default configurations are provided for optional settings
- User inputs in forms and UI components include sanitization logic
- Module boundaries have explicit type contracts and validation for configuration data flow
- Validation failures produce clear, actionable error messages with context
- No runtime errors occur due to invalid, missing, or malformed configuration values
- Configuration requirements are self-documenting through validation logic

<enforcement>
Claude Code MUST NOT skip or defer validation implementation when working with configuration, environment variables, or user inputs. Defensive validation at boundaries is mandatory to prevent runtime errors and security vulnerabilities.
</enforcement>
