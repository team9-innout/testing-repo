<rule_activation id="7f1c0679-de17-4a67-bd4d-5d353947917e" title="Adopt Environment-Based Configuration for External API Integration in Client-Side Applications" applies_to="apps/**">
These rules are ALWAYS ACTIVE for all files matching `apps/**` (client-facing applications including documentation site and demo app).
</rule_activation>

### Rules

- **R-ENV-001** MUST: All external API integrations in client-facing applications must use environment variables for configuration (API keys, endpoints, credentials).
- **R-ENV-002** MUST: API keys and sensitive credentials must never be hardcoded in source code.
- **R-ENV-003** MUST: Secrets and API keys must never be committed to version control.
- **R-ENV-004** MUST: Configuration must be initialized at the application root level in document/layout/theme configuration files (_document.tsx, layout.tsx, theme.config.tsx).
- **R-ENV-005** MUST: Environment variables must be properly scoped to server-side or build-time contexts where appropriate.
- **R-ENV-006** MUST: Configuration must be environment-specific (development, staging, production).
- **R-ENV-007** SHOULD: Use build-time environment variable injection for optimization and dead code elimination where appropriate.
- **R-ENV-008** SHOULD: Provide environment variable documentation and example configurations (.env.example) for local development setup.
- **R-ENV-009** MAY: Expose certain API keys client-side only after careful evaluation of security implications for specific services that require it.

### Verify

```bash
# Check for hardcoded API keys or secrets in source code
grep -rE '(api[_-]?key|secret|password|token)\s*[=:]\s*["\x27][A-Za-z0-9_-]{20,}' apps/ --include="*.tsx" --include="*.ts" --include="*.jsx" --include="*.js" | grep -v "process.env" | grep -v ".env.example" || echo "✓ No hardcoded secrets found"

# Verify configuration is centralized in layout/document/theme files
find apps/ -type f \( -name "_document.tsx" -o -name "layout.tsx" -o -name "theme.config.tsx" \) | while read file; do
  if grep -q "process.env" "$file"; then
    echo "✓ $file uses environment variables"
  else
    echo "⚠ $file may need environment variable integration"
  fi
done

# Check that .env files are properly gitignored
if git check-ignore -q .env .env.local .env.*.local 2>/dev/null || grep -qE '^\.env(\.local)?$' .gitignore 2>/dev/null; then
  echo "✓ .env files are gitignored"
else
  echo "✗ .env files should be added to .gitignore"
fi

# Verify .env.example exists for developer onboarding
if [ -f .env.example ] || [ -f apps/.env.example ]; then
  echo "✓ .env.example found for documentation"
else
  echo "⚠ Consider adding .env.example for developer onboarding"
fi
```

**Accept when:**
- No hardcoded API keys or secrets are present in source code (apps/**/*.{ts,tsx,js,jsx})
- All external API configurations use environment variables (process.env.VARIABLE_NAME)
- Configuration is centralized in application layout/document/theme files
- Secrets are not committed to version control (.env files are gitignored)
- Environment-specific configurations are properly managed across development, staging, and production
- Runtime environment variable injection is used for secrets management

<enforcement>
Claude Code MUST NOT skip or defer verification. When modifying files in apps/**, verify that environment-based configuration patterns are followed and no secrets are hardcoded or committed.
</enforcement>
